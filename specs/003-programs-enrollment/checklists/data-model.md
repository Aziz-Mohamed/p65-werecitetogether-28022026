**Created**: 2026-03-04
**Feature**: [spec.md](../spec.md)
**Status**: All 42 items resolved (2026-03-04)

## Requirement Completeness

- [x] CHK001 Are ON DELETE behaviors explicitly specified for all 12 foreign keys across the 5 tables? [Completeness, Data-Model §cohorts — `teacher_id` is NOT NULL but ON DELETE SET NULL would leave orphaned NOT NULL violation] → **RESOLVED**: All 12 FKs now have explicit ON DELETE in data-model.md FK tables. `cohorts.teacher_id` uses ON DELETE RESTRICT (not SET NULL) to prevent NOT NULL violation.
- [x] CHK002 Is the `programs` table missing an English `name` column requirement in the spec, or is bilingual naming only specified narratively without column-level definition? [Completeness, Spec §FR-001] → **RESOLVED**: `name` (English) and `name_ar` (Arabic) columns are explicitly defined in data-model.md. FR-001 says "bilingual name and description" which maps to these columns.
- [x] CHK003 Are default values for `settings` JSONB keys (`max_students_per_teacher`, `auto_approve`, `session_duration_minutes`) specified with explicit defaults for the 8 seeded programs? [Completeness, Spec §FR-029, Gap] → **RESOLVED**: Added defaults to spec Assumptions (10, false, 30). Seed data in data-model.md specifies per-program overrides (Program 3: `auto_approve: true`).
- [x] CHK004 Is the `curriculum` JSONB column on `program_tracks` defined with a schema or structure requirement, or left as opaque blob? [Completeness, Data-Model §program_tracks] → **RESOLVED**: Added `TrackCurriculum` TypeScript interface to data-model.md JSONB Schemas section (units, total_duration_weeks, reference_text). Noted as optional/informational. Updated FR-003 in spec.
- [x] CHK005 Are `updated_at` trigger requirements specified for all 5 tables, including whether the existing `handle_updated_at` function is reused or a new one created? [Completeness, Data-Model §programs, §program_tracks, §cohorts, §enrollments] → **RESOLVED**: Each table now explicitly states "Reuses existing `handle_updated_at()` function from migration 00001." `program_roles` has no `updated_at` column (immutable — delete and recreate).
- [x] CHK006 Is the `schedule` JSONB column on `cohorts` defined with a structural requirement beyond the single example `[{"day": 0, "start": "18:00", "end": "19:00"}]`? [Clarity, Spec §Assumptions] → **RESOLVED**: Added `CohortScheduleEntry` TypeScript interface to data-model.md. Updated spec Assumptions with day=0-6, HH:mm format definition.
- [x] CHK007 Are requirements specified for what happens to `program_roles` when a profile's top-level `role` in `profiles` changes (e.g., teacher demoted to student)? [Gap, Edge Case] → **RESOLVED**: Added "Profile role demotion" edge case to spec. `program_roles` entries remain until explicitly removed by admin. Route-group access is lost via role check, but records preserved for audit.
- [x] CHK008 Is there a requirement for an `enrollment_count` materialized/computed column or view on `cohorts`, or is count always derived via query? [Completeness, Spec §FR-017] → **RESOLVED**: Documented in data-model.md cohorts section — count derived via query with `idx_enrollments_cohort_id` index. Materialization deferred unless scale demands it.

## Requirement Clarity

- [x] CHK009 Is the enrollment status `approved` vs `active` distinction clearly defined? The spec §FR-008 lists both but the state transition diagram and `enroll_student()` function only use `active` — is `approved` a synonym or a distinct intermediate state? [Ambiguity, Spec §FR-008, Data-Model §State Transitions] → **RESOLVED**: Removed `approved` status entirely. 5 statuses remain: `pending`, `active`, `completed`, `dropped`, `waitlisted`. Admin approval transitions `pending` → `active` directly. Updated spec FR-008, data-model CHECK constraint, and state transition diagram.
- [x] CHK010 Is "auto-resolve pending enrollments" on cohort `in_progress` transition quantified — are ALL pending students approved, or only those meeting criteria? What happens to waitlisted students? [Clarity, Spec §US5-AS4] → **RESOLVED**: Updated US5-AS4 — all `pending` enrollments bulk-approved to `active`. Waitlisted students remain `waitlisted` (manual promotion by admin, or deferred to spec 006).
- [x] CHK011 Is the `COALESCE` nil-UUID strategy for the unique enrollment index clearly specified as the chosen approach for nullable unique constraints, vs partial indexes? [Clarity, Data-Model §enrollments] → **RESOLVED**: Added detailed explanation in data-model.md with rationale for COALESCE over partial indexes. Documented that PostgreSQL treats `(A, NULL) != (A, NULL)` in regular UNIQUE.
- [x] CHK012 Is the `get_user_programs()` return behavior specified for users with NO program associations (new user, master_admin with no explicit roles)? [Clarity, Data-Model §Helper Functions] → **RESOLVED**: Added note — returns empty array `'{}'::uuid[]`. Master admin access uses role check, not this function. New students see no program-scoped data until they enroll.
- [x] CHK013 Is "mixed" category behavior precisely defined at the data model level — does a mixed program's free tracks skip cohort requirements while structured tracks require them? [Ambiguity, Spec §Context, Gap] → **RESOLVED**: Added `track_type` column to `program_tracks` (`free` or `structured`, nullable). Mixed programs' tracks have explicit types. `enroll_student()` uses track_type to determine enrollment path. Updated FR-003, FR-029, and data-model.
- [x] CHK014 Are the exact conditions under which `enroll_student()` handles mixed-category programs specified? The function branches on `category = 'free'` but mixed programs have both free and structured sections. [Ambiguity, Data-Model §enroll_student()] → **RESOLVED**: Rewrote `enroll_student()` with `v_effective_type` logic — checks `track_type` for mixed programs. No track in mixed program defaults to free. Fully specified in data-model.md.

## Requirement Consistency

- [x] CHK015 Does the `cohorts.teacher_id` NOT NULL constraint conflict with the creation flow where a cohort might initially lack a teacher assignment? [Conflict, Data-Model §cohorts vs Spec §US5-AS1 — teacher_id listed as required] → **RESOLVED**: `teacher_id` NOT NULL is correct (US5-AS1 requires it). Changed ON DELETE from SET NULL to RESTRICT — prevents deleting teacher profile while assigned to cohort. Admin must reassign first.
- [x] CHK016 Is the enrollments INSERT RLS policy consistent between direct free-program inserts and `enroll_student()` RPC? The RPC uses `SECURITY DEFINER` (bypasses RLS) while free joins use direct insert (requires INSERT policy). [Consistency, Data-Model §enrollments RLS, Contracts §PR-006 vs §PR-007] → **RESOLVED**: Documented both paths explicitly. INSERT policy added for `student_id = auth.uid() AND get_user_role() = 'student'`. RPC bypasses RLS for structured. Updated contracts PR-007.
- [x] CHK017 Does the `programs` UPDATE RLS using `program_id = ANY(get_user_programs())` make sense for the `programs` table itself? The `programs` table has `id` not `program_id` — the policy should use `id = ANY(...)`. [Conflict, Data-Model §programs RLS] → **RESOLVED**: Fixed to `id = ANY(get_user_programs())`. Added explicit note in data-model.md.
- [x] CHK018 Is the enrollment unique constraint `(student_id, program_id, track_id, cohort_id)` consistent with the free-program enrollment where both `track_id` and `cohort_id` are NULL — does this allow only one free enrollment per program? [Consistency, Spec §FR-009, Data-Model §enrollments] → **RESOLVED**: Yes, COALESCE maps NULLs to sentinel UUID, allowing exactly one free enrollment per student per program. Documented explicitly in data-model.md.
- [x] CHK019 Do the `enroll_student()` capacity-count statuses (`pending`, `active`, `approved`) align with all upstream references to capacity-consuming statuses? [Consistency, Data-Model §enroll_student() line 239 vs Spec §FR-012] → **RESOLVED**: After removing `approved`, capacity-consuming statuses are `pending` and `active` (2 statuses). Consistent across RPC, FR-012, and enrollment count strategy.
- [x] CHK020 Is the `program_tracks` ON DELETE CASCADE on `program_id` consistent with the requirement that tracks with active enrollments cannot be deleted? [Conflict, Spec §FR-006, Data-Model §program_tracks] → **RESOLVED**: Changed to ON DELETE RESTRICT. Programs with tracks cannot be deleted at DB level. Deactivation (`is_active = false`) is the correct workflow per FR-006.

## Edge Case & Failure Handling Coverage

- [x] CHK025 Are requirements defined for `enroll_student()` behavior when `auth.uid()` returns NULL (unauthenticated call)? [Edge Case, Data-Model §enroll_student(), Gap] → **RESOLVED**: Added `auth.uid() IS NULL` check at top of function — raises `'Authentication required'`. Added FR-033a to spec.
- [x] CHK026 Are requirements specified for what happens when a program is deactivated while `enroll_student()` is mid-transaction (race condition between deactivation and enrollment)? [Edge Case, Spec §Edge Cases, Gap] → **RESOLVED**: Added "Program deactivation during enrollment" edge case to spec. RPC's `WHERE is_active` check prevents enrollment of deactivated programs.
- [x] CHK027 Is the behavior specified when a student's enrollment references a cohort whose track is later deactivated (`track_id` FK still valid but track `is_active = false`)? [Edge Case, Gap] → **RESOLVED**: Added "Deactivated track with existing enrollments" edge case to spec. Existing enrollments remain valid; track stops appearing in browsing UI.
- [x] CHK028 Are requirements defined for the `enroll_student()` error message format and whether errors are user-facing strings or internal codes? [Clarity, Data-Model §enroll_student(), Gap] → **RESOLVED**: RPC uses structured error codes (`ENROLL_PROGRAM_NOT_FOUND`, etc.). Service layer maps to i18n strings. Updated data-model.md and contracts PR-006.
- [x] CHK029 Is the waitlist position calculation requirement specified? Spec §US2-AS3 says "waitlist position" but no ordering column exists on enrollments. [Gap, Spec §US2-AS3] → **RESOLVED**: Added to spec Assumptions — position derived from `enrolled_at` FIFO. Added calculation formula to data-model.md enrollments section.
- [x] CHK030 Are requirements specified for cascade behavior when a `programs` row is deleted — all tracks, cohorts, enrollments, and program_roles would cascade-delete. Is this intentional given the no-deletion philosophy? [Edge Case, Data-Model §programs — no ON DELETE specified] → **RESOLVED**: `programs` child FKs use ON DELETE RESTRICT (tracks, cohorts, enrollments) — programs cannot be deleted if they have any children. Only `program_roles` uses CASCADE (safe because programs are RESTRICT-protected by other FKs).

## RLS & Security Requirements

- [x] CHK031 Are RLS requirements specified for the `supervisor` role's read access to enrollments within their assigned programs? [Gap, Spec §FR-030–033 only mention student/program_admin/master_admin] → **RESOLVED**: Added FR-030b — supervisors can read all enrollments within their assigned programs. Added supervisor SELECT policy to data-model.md enrollments RLS.
- [x] CHK032 Are RLS requirements specified for `teacher` role's read access to enrollments where they are the assigned teacher? [Gap, Spec §FR-030–033] → **RESOLVED**: Added FR-030a — teachers can read enrollments where they are assigned teacher or belong to their cohort. Added teacher SELECT policy to data-model.md.
- [x] CHK033 Is the `get_user_programs()` SECURITY DEFINER function requirement documented with threat analysis — can a malicious user exploit it to escalate program access? [Gap, Security] → **RESOLVED**: Added security note to data-model.md — function only returns program_ids (not row data), filters by `auth.uid()`, and is STABLE (read-only). Cannot be exploited for access escalation.
- [x] CHK034 Are RLS requirements for the `enroll_student()` function specified — specifically, is it documented that the function must validate the caller is a student role? [Clarity, Data-Model §enroll_student()] → **RESOLVED**: Added student role check to RPC. Added FR-033a to spec. RPC now validates `auth.uid()` and student role before processing.
- [x] CHK035 Is there a requirement preventing program admins from assigning themselves to additional programs via `program_roles` INSERT policy? [Gap, Security, Data-Model §program_roles RLS] → **RESOLVED**: INSERT policy includes `profile_id != auth.uid()` for program_admin role. Updated FR-031 in spec and program_roles RLS in data-model.md.
- [x] CHK036 Are RLS requirements specified for the `admin` role (legacy) — does it have any access to program data or is it excluded? [Gap, Spec §FR-030–033] → **RESOLVED**: Added FR-032a — legacy admin has NO access to program data. Added note to enrollments RLS in data-model.md.
